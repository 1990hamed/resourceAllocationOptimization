# Hospital Resource Allocation Optimization

A comparative study of **Grey Wolf Optimizer (GWO)** vs **Particle Swarm Optimization (PSO)** for hospital resource allocation. The pipeline runs end-to-end from raw data ingestion through statistical reporting.

## Problem Statement

Hospitals must simultaneously decide:

- **Staff scheduling** — how many surgeons, nurses, and technicians to assign across shifts, subject to minimum coverage requirements and a per-shift budget cap.
- **Inventory ordering** — how much of each medical supply to reorder, balancing restock costs against holding costs and stockout risk.

The objective minimises total cost (staffing + overtime + restock + holding) plus a penalty for hard constraint violations (under-staffing, over-budget shifts, stockouts).

## Algorithms

| Algorithm | Reference | Key parameters |
|---|---|---|
| Grey Wolf Optimizer (GWO) | Mirjalili et al., 2014 | `a` linearly decays 2 → 0; three-leader hierarchy (α, β, δ) |
| Particle Swarm Optimization (PSO) | Kennedy & Eberhart, 1995 | Inertia `w` decays 0.9 → 0.4; `c1 = c2 = 2.0` |

Both run for **200 iterations** with a **population of 30** over **30 independent runs** (seeds 42–71) for statistically robust comparison.

## Quick Start

```powershell
# 1. Install dependencies
uv sync

# 2. Run the full pipeline end-to-end
uv run main.py --all
```

Or run each step individually:

```powershell
uv run main.py --step audit        # Data quality checks
uv run main.py --step preprocess   # Clean and transform raw data
uv run main.py --step eda          # Exploratory analysis + EDA plots
uv run main.py --step optimize     # GWO vs PSO experiment + comparison plots
uv run main.py --step report       # Build Jupyter report notebook
```

## Fitness Function

```
fitness = total_cost + penalty_weight × total_violation
```

**Cost components:**
- Regular staffing cost per shift (`staff_count × hours × hourly_rate`)
- Overtime cost at 1.5× for hours beyond the 8-hour threshold
- Inventory restock cost (`order_qty × unit_cost`)
- Inventory holding cost (`order_qty × unit_cost × 0.02% daily`)

**Penalty terms** (weight = 1 × 10⁶):
- Under-staffing: `max(0, min_staff − allocated)` per shift per staff type
- Over-budget: `max(0, daily_cost − budget_limit)` per shift
- Stockout: `max(0, min_order − allocated)` per inventory item

## Configuration

All hyperparameters are centralised in [src/rao/config.py](src/rao/config.py):

| Constant | Default | Description |
|---|---|---|
| `RANDOM_SEED` | 42 | Base seed; runs use seeds 42, 43, … |
| `N_RUNS` | 30 | Independent runs per algorithm |
| `MAX_ITER` | 200 | Iterations per run |
| `POP_SIZE` | 30 | Population size |
| `PENALTY_WEIGHT` | 1e6 | Hard constraint violation penalty |
| `OVERTIME_MULTIPLIER` | 1.5 | Overtime pay multiplier |
| `HOLDING_COST_FACTOR` | 0.0002 | Daily holding cost (~2% annual) |

## Statistical Analysis

After the optimize step, `rao.experiments.stats` produces:

- **Wilcoxon signed-rank test** (paired, one-sided: GWO < PSO)
- **Mann-Whitney U test** (unpaired robustness check)
- **Rank-biserial correlation** (effect size `r`)
- Per-iteration convergence statistics across all runs
